Lokaler Start

python -m venv .venv
. .venv/bin/activate
pip install -r requirements-dev.txt
pre-commit install
npm install
python src/manage.py migrate
python src/manage.py bootstrap_roles
python src/manage.py runserver

Beim ersten Aufruf führt die App durch die Ersteinrichtung und legt den ersten Admin an.

Konfiguration

Alle produktionsrelevanten Platzhalter stehen in .env.example.

  • DJANGO_SECRET_KEY: produktiver Secret Key.
  • DJANGO_DEBUG: 1 lokal, 0 für Docker/Deployment.
  • DJANGO_ALLOWED_HOSTS: kommaseparierte Hostnamen.
  • CSRF_TRUSTED_ORIGINS: kommaseparierte Origins inklusive Schema.
  • DATABASE_URL: Datenbank-URL; Docker nutzt PostgreSQL, E2E nutzt SQLite.

Docker

cp .env.example .env
docker compose up --build

Danach läuft die App unter http://localhost:8000. Der Compose-Start führt Migrationen, bootstrap_roles, collectstatic und anschließend Gunicorn aus.

Typischer Admin-Ablauf

  1. Lager anlegen und Lager-Fördersatz pflegen.
  2. Nutzer und Rollen in der Nutzerverwaltung pflegen.
  3. Preisverwaltung öffnen und Lagerpauschalen für Teilnehmer/Begleiter und 1/2 Wochen eintragen.
  4. Getränkepreise, Standardpreise für Frühstück/Abendessen und bei Bedarf abweichende Tagespreise anlegen.
  5. Teilnehmer importieren oder manuell anlegen.
  6. PINs setzen oder Teilnehmer im Kiosk ihre PIN beim ersten Login setzen lassen.
  7. Während des Lagers Kiosk für Buchungen nutzen: Getränke per großer Ein-Tap-Taste buchen, Essen anmelden, Countdown zur automatischen Abmeldung im Blick behalten.
  8. Falsche Buchungen als Admin korrigieren und Audit-Protokoll prüfen.
  9. Abrechnung prüfen und Exporte erzeugen: Lager-CSV, Getränke-CSV, Excel-Arbeitsmappe und Einzel-PDF.

Import und Export

Der Teilnehmerimport akzeptiert CSV und XLSX. Pflichtspalten sind first_name und last_name; weitere Spalten sind email, phone, status, is_child, is_youth_group, is_companion, hilfssatz, berufssatz, booked_nights, actual_nights und notes.

Die Importseite zeigt eine Vorschau. Beim Speichern werden nur gültige Zeilen übernommen; vorhandene Teilnehmer werden pro Lager anhand von Vor- und Nachname aktualisiert.

Tests

.venv/bin/python src/manage.py check
.venv/bin/python -m pytest
.venv/bin/python -m ruff check .
.venv/bin/python -m ruff format --check .
.venv/bin/python -m mypy src
npm run test:e2e
npm run test:local

Die Tests decken Authentifizierung, Rollen, Nutzerverwaltung, Import, Export, Abrechnung, Kiosk, Preisverwaltung, Buchungs-Audit und View-Berechtigungen ab. npm run test:local schreibt Logs nach .test-local-logs/<timestamp>/.

CI/CD & Automatisierung

Das Projekt nutzt GitHub Actions (.github/workflows/) für kontinuierliche Integration:

  • ci.yml: Führt bei jedem Push und PR lokale Tests und Playwright aus.
  • docker.yml: Baut das Image, testet es und pusht es nach einem Merge in die GitHub Container Registry (ghcr.io).
  • security.yml: Führt regelmäßige Trivy Vulnerability Scans durch.
  • pr-title.yml & changelog-check.yml: Erzwingen saubere PR-Namen und Changelog-Einträge.
  • dependabot.yml: Automatisiert Abhängigkeits-Updates.

Qualitäts- und Sicherheitsregeln

  • Ruff erzwingt Linting und Formatierung.
  • pre-commit führt lokale Checks und Secret-Scanning aus.
  • Neue Python-Funktionen sollen Type Hints nutzen.
  • .env-Dateien und echte Secrets dürfen nicht ins Repository.
  • Abrechnungs- und Kiosk-Operationen sollen transaktional und ORM-effizient umgesetzt werden.
  • Templates sollen semantisches HTML, mobile-first CSS und barrierearme Formulare nutzen.
  • Dynamisches Verhalten soll mit modernem Vanilla JavaScript oder bewusst ausgewählten leichten Tools umgesetzt werden.
